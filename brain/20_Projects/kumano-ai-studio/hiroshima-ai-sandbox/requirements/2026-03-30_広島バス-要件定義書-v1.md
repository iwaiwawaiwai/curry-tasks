## 1. プロジェクト概要

### 背景・目的

広島バスには既存サービス「来るけん」があるが、「たどり着けない人が多い」という課題がある（特にご年配の乗客）。HPへの誘導が弱く、定期代・バス停・遅延に関する問い合わせが多発している。

熊野AIスタジオが4日間で開発したプロトタイプは「来るけんのサクサク動く・見やすいバージョン」として採用検討に進んでいる。このプロトタイプを本番サービスとして運用できる状態に引き上げることが本プロジェクトの目的。

### 解決する課題

| # | 課題 | 広島バスへの影響 |
|---|------|----------------|
| 1 | アプリへの誘導が弱い（来るけんへのアクセス困難） | 乗客がコールセンターに頼らざるを得ない |
| 2 | 問い合わせが多い（定期代・遅延・バス停） | 電話対応コストが発生 |
| 3 | 遅延証明書の発行フローが未整備 | 担当者が手作業で対応 |
| 4 | 制服貸与管理がデジタル化されていない | 内部業務の手間が発生 |

### 期待する効果

| 効果の種類 | 内容 |
|-----------|------|
| コスト削減 | 問い合わせ対応工数を削減（★件数・コストは広島バスから収集中） |
| 業務効率化 | 遅延証明書・制服管理のデジタル化で担当者の手間を削減 |
| ブランド向上 | 「迷わないバス」としての乗客体験向上 |

## 2. 提供サービスの範囲（スコープ）

### 対象システム一覧（5システム）

| # | システム名 | 乗客/社内 | 一言で言うと |
|---|-----------|---------|-------------|
| 1 | **バスどこ** | 乗客向け | リアルタイムでバスの位置・遅延を確認するWebアプリ |
| 2 | **GTFS-RT分析** | 社内向け | 遅延傾向・路線パフォーマンスを可視化するダッシュボード |
| 3 | **制服貸与管理** | 社内向け | 従業員の制服の貸出・返却をPC上で管理するツール |
| 4 | **運賃距離証明書** | 乗客向け | 乗車区間の運賃・距離をPDFで発行するWebツール |
| 5 | **遅延証明書** | 乗客向け | 遅延が発生した際の証明書をPDFで発行するWebツール（新規開発） |

### 各システムの主要機能

**① バスどこ（乗客向けWebアプリ）**
- バス停検索 → 通過便一覧・接近バス表示
- リアルタイム位置情報・遅延情報（5分以上：黄、10分以上：赤）
- 時刻表・料金・定期代表示
- お気に入りバス停登録
- QRコードでシェア対応
- データソース：GTFS-RT（30秒更新）、来るけん

**② GTFS-RT分析ダッシュボード（社内向け）**
- 遅延傾向の可視化（路線別・時間帯別・天候別）
- 路線パフォーマンスランキング・ヒートマップ
- データ蓄積・日次集計（raw 7日保持 → 集計データ永続保存）

**③ 制服貸与管理（社内ツール・ローカル運用）**
- 従業員の制服貸与・返却記録（401人対応）
- CSV管理・PC上でHTMLファイルを直接開く運用
- サーバー不要・Chrome/Edge限定

**④ 運賃距離証明書（PDF発行）**
- 区間・バス停間の運賃・距離をPDFで発行
- 完全静的サイト・バックエンド不要

**⑤ 遅延証明書（新規開発）**
- 遅延発生時の証明書をPDFで発行
- ④のコードベースを流用して新規開発

### スコープ外
- 来るけん本体システムの改修・保守
- チケット購入・決済機能
- ハードウェアの設置・管理
- 制服管理のWeb化（必要になれば別途追加開発）

## 3. 利用者・利用シナリオ

### 利用者区分

| 利用者 | 対象システム | 特性 |
|--------|------------|------|
| バス乗客（一般） | バスどこ・証明書2種 | スマホでQRアクセス。特にご年配への配慮が必要 |
| 広島バス 現場担当者 | 制服管理・GTFS-RT分析 | 社内PC（Chrome/Edge）で利用 |
| 広島バス 管理職 | GTFS-RT分析 | 遅延状況・路線改善の判断材料として活用 |

### 主要ユースケース

1. 乗客が「次のバスはいつ来る？何分遅れてる？」をリアルタイムで自己解決する
2. 乗客が「定期代はいくら？」「運賃はいくら？」をPDFで取得する
3. 乗客が遅延に遭い、会社や学校への提出用に遅延証明書を発行する
4. 担当者が遅延が多い路線・時間帯を把握して運行改善の判断に使う
5. 担当者が制服の貸出・返却状況をPCで管理する

## 4. システム・技術要件

### インフラ構成

```
[Vercel Pro（約3,000円/月）]
├── バスどこ フロントエンド + GTFS-RTプロキシ
├── 運賃距離証明書（静的サイト）
└── 遅延証明書（静的サイト）

[VPS さくら or ConoHa 2GB（約1,200〜1,800円/月）]
├── バスどこ collector（Docker）— 30秒ポーリング
├── GTFS-RT分析 collector + API（Docker）
├── SQLite DB
└── 日次バックアップ

[クライアント端末（Chrome/Edge）]
└── 制服貸与管理（ローカルHTML + CSV）
```

### 月額ランニングコスト（インフラのみ）

| 項目 | 月額 |
|------|------|
| VPSサーバー | 1,200〜1,800円 |
| Vercel Pro | 約3,000円 |
| ドメイン | 100〜250円 |
| 監視（UptimeRobot / Sentry） | 0円（無料枠） |
| **合計** | **約4,300〜5,550円** |

※ AI API利用料（Claude等）は含まない

### 動作環境
- **バスどこ・証明書系**：スマートフォン（iOS Safari・Android Chrome）対応のWebアプリ
- **制服管理**：社内PC Chrome/Edge 限定（File System Access API依存）
- **GTFS-RT分析**：Webブラウザ（社内アクセス制限付き）

### セキュリティ方針
- HTTPS強制（Let's Encrypt + Vercel自動）
- GTFS-RT分析：IP制限 or Basic認証でアクセス制限
- 制服管理：従業員データはローカル端末内のみ（外部サーバーに送信しない）
- ★GTFS-RT APIの商用利用規約は確認が必要

## 5. 運用・サービスレベル（SLA）

### 稼働時間・障害対応

| 項目 | 方針（案） |
|------|-----------|
| 稼働時間 | 24時間365日（バスどこcollectorはJST 5:00〜24:00稼働） |
| 障害時の初動 | **平日9:00〜18:00の4時間以内に対応開始**（一人会社のため24/365は現実的でない） |
| バスどこcollector停止時 | リアルタイム表示が止まるが人命に関わらない。速やかに復旧対応 |
| 計画メンテナンス | 深夜2〜5時（バス運行なし時間帯）で実施 |

### 監視体制
- 死活監視：UptimeRobot（5分間隔・無料枠）
- エラー監視：Sentry（月5,000イベントまで無料）
- アラート：ダウン検知 → メール or Slack通知

### ★未確認事項
- 広島バス側が求めるSLAの水準
- 障害時の連絡ルートと担当者

## 6. 開発・本番化スケジュール（案）

| フェーズ | 経過期間 | 主な作業 |
|----------|---------|---------|
| Phase 1: 基盤構築 | 2〜3週間 | VPS契約・Docker環境・ドメイン・SSL |
| Phase 2: 開発・テスト | 4〜6週間 | 遅延証明書の新規開発・セキュリティ設定・各種テスト |
| Phase 3: データ移行・検証 | 2〜4週間 | GTFS-RTデータ蓄積・負荷テスト・UAT |
| Phase 4: リリース | 1〜2週間 | 本番デプロイ（段階的）・監視設定・保守契約締結 |
| Phase 5: 安定化 | 1〜2週間 | 本番後の初期対応・運用フロー確認 |
| **合計** | **約4〜5ヶ月** | データ蓄積待ち・クライアント応答待ち含む |

★詳細スケジュールは要件確定後に森くんと更新

| システム | 工数（標準） | 開発費（@5万/日） |

| 制服貸与管理 | 5〜8日 | 25〜40万 |
| 運賃距離証明書 | 5〜7日 | 25〜35万 |
| 共通作業 | 7〜9日 | 35〜45万 |
| **合計** | **67〜92人日** | **335〜460万** |

※ AI活用（Claude Code等）でコーディング工数は圧縮済み。データ検証・UAT・クライアント調整は従来通りの工数。

### 月額保守費（案）

| 項目 | 金額（案） | 根拠 |
|------|-----------|------|
| インフラ実費 | 約4,300〜5,550円 | VPS + Vercel Pro + ドメイン |
| 保守・改修工数 | ★別途決定 | 月間対応工数の目安で設定 |
| **月額合計（案）** | **15〜20万円** | くるけん相当（約16万円）と同等水準で「より高機能」を訴求 |

### 費用根拠（クライアントへの説明ロジック）

業界水準：くるけん相当サービスの月額は3,000〜4,000円/台 × 400台 ≈ 月16万円。
本サービスは乗客UX・社内業務効率化・データ分析機能を追加した上で同等価格帯に設定。
★削減コストの定量把握後、費用対効果の根拠として提示する。

### 支払い条件（案）
- 初期費用：契約後2週間以内に一括
- 月額：月末締め翌月払い

### 契約期間（案）
- 最低契約期間：12ヶ月
- 自動更新（3ヶ月前に解約申し出がない場合）

## 9. 前提条件・制約

### 広島バス側に求める協力事項

| # | 内容 |
|---|------|
| 1 | GTFS-RT APIの商用利用承認（または利用規約確認の協力） |
| 2 | ★月間問い合わせ件数・対応コストの数値提供 |
| 3 | ★料金・定期代・遅延証明書の表記内容確定 |
| 4 | ★IT環境確認（ブラウザ・ネットワーク・セキュリティポリシー） |
| 5 | UAT担当者の確保（Phase 3で必要） |
| 6 | ドメインの取り扱い方針（広島バス名義 or 熊野AIスタジオ名義） |

### リスクと対策

| リスク | 影響度 | 対策 |
|--------|--------|------|
| GTFS-RT公開APIの仕様変更・停止 | 高 | 利用規約確認・停止時のフォールバックUI設計 |
| VPS collector停止（メモリ不足等） | 高 | Docker restart: always・メモリ監視 |
| 広島バス側のUAT遅延 | 中 | スケジュール確定時に早めにUAT日程を押さえる |
| 広島AIサンドボックス補助金の知財条件 | 中 | ★補助金事務局に条件確認・契約前に合意 |
| 遅延証明書の仕様変更リスク | 中 | 仕様を先に確定してから開発開始 |
